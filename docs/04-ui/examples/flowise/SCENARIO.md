# Flowise シナリオ撮影計画

このファイルは [../03-flowise.md](../03-flowise.md) の証跡を、本文の意味と一致する順序で取得するための操作台本です。

## 目的

- 各画像が本文の該当手順をそのまま証明すること
- 空画面、ローディング中、意味のずれた代替画像を採用しないこと
- 再撮影時も同じ結果を再現できること

## 前提条件

- Flowise コンテナが `Up` である
- 管理者アカウント作成済み、または初回セットアップ画面まで到達できる
- Provider は OpenAI または Ollama のどちらか一方で事前に接続方針を決める
- 実行用のテキスト: `こんにちは。3行で自己紹介して。`

## シナリオ本体

### 01. コンテナ起動確認

- 目的: Flowise が起動済みでポート 3001 を待受していることを示す
- 操作: `docker compose ps` と `docker ps --filter name=flowise` を取得する
- 採用条件:
  - `flowise` の `Up` が見える
  - `0.0.0.0:3001->3000/tcp` が見える
- 不採用条件:
  - `Created` や `Restarting` のまま
  - 出力の大半が余白

### 02. 初回セットアップ画面

- 目的: 初回アクセス直後の管理者作成画面を示す
- 操作: `http://localhost:3001` にアクセスする
- 採用条件:
  - `Setup Account` 見出しが見える
  - Name / Email / Password の入力欄が見える
- 不採用条件:
  - ローディング中
  - 画面上部だけで主入力欄が切れている

### 03. Chatflow 作成画面

- 目的: Chatflow に必要なノードが実際に配置された状態を示す
- 操作:
  1. `Add New` をクリック
  2. `Prompt Template` を追加
  3. `OpenAI` または `Ollama` の Chat Model を追加
  4. `LLM Chain` を追加
  5. `Prompt Template -> LLM Chain (Prompt)`、`Chat Model -> LLM Chain (Language Model)` を接続
  6. 3ノードを同時に視認できる位置に並べる
- 採用条件:
  - 3つのノード名（Prompt Template / OpenAI(or Ollama) / LLM Chain）が読める
  - キャンバス中央にノードがあり、余白が主役でない
- 不採用条件:
  - 必須ノードが不足している
  - Add Nodes パネルだけでキャンバスが見えない

### 04. Provider 設定画面

- 目的: 実行に必要な Provider 設定欄が見えていることを示す
- 操作:
  1. Chat Model ノードを開く
  2. `Connect Credential` または `Base URL` を表示する
- 採用条件:
  - Provider 名が読める
  - 資格情報入力欄または接続欄が見える
- 不採用条件:
  - 単なるノード一覧のみ
  - 設定欄が折りたたまれて読めない

### 05. テスト入力画面

- 目的: 実行前に入力テキストが入っている状態を示す
- 操作:
  1. Prompt Template の Human Message に `こんにちは。3行で自己紹介して。` を入力
  2. 実行ボタンを押す前に撮影する
- 採用条件:
  - 入力テキストの全部または主文が見える
  - 実行前の状態である
- 不採用条件:
  - 入力欄が空
  - ローディングモーダルが前面に出ている

### 06. テスト出力画面

- 目的: 実行結果が返ったことを示す
- 操作:
  1. 右下チャット実行パネルから質問を送信する
  2. 応答テキストまたは明確な実行エラーが表示されるまで待つ
  3. 結果パネルが見えた状態で撮影する
- 採用条件:
  - 応答文、または原因追跡可能なエラー文が見える
  - 実行後の状態だと判断できる
- 不採用条件:
  - ローディング中
  - 05 と同じ画面
  - 単なる設定画面の代用

## 品質ゲート

1. 各画像に「何を証明する画像か」を1文で説明できること
2. 本文の見出し名と画像の意味が一致すること
3. 主対象が画像の中心にあり、余白が主役でないこと
4. 05 と 06 は必ず別状態であること
5. 不採用条件に当てはまる画像を代替採用しないこと

## リトライルール

1. ローディング中を撮った場合は不採用として撮り直す
2. 対象 UI が閉じている場合は再度その UI を開いてから撮る
3. 画面内情報が不足する場合は、ズームやパネル開閉で情報量を増やしてから再撮影する
4. 06 が取得できない場合は Provider 設定不備として run-log に残し、代替画像で埋めない

## 完了判定

- `01` から `06` のすべてで採用条件を満たす
- `run-log.txt` の結果欄が実態と一致する
- [../03-flowise.md](../03-flowise.md) の手順と画像の意味が一致する
