# LibreChat シナリオ撮影計画

このファイルは [../04-librechat.md](../04-librechat.md) の証跡を、本文の意味と一致する順序で取得するための操作台本です。

## 目的

- 各画像が本文の該当手順をそのまま証明すること
- 05 と 06 を必ず別状態で取得すること
- 通常応答を疎通確認、Tool Call / MCP を価値確認として切り分けること
- Tool Call / MCP を有効にした構成では、その実行結果を優先して 06 に採用すること

## 前提条件

- `sandbox/LibreChat` にリポジトリが展開されている
- `.env` に `HOST=0.0.0.0`、`PORT=3080`、利用 Provider の API キーが設定されている
- `ALLOW_EMAIL_LOGIN=true`、`ALLOW_REGISTRATION=true` を確認済み
- 初回起動は pull に時間がかかるため、Web UI が安定表示するまで待つ

## シナリオ本体

### 01. `.env` 作成直後

- 目的: `.env.example` から `.env` を作成したことを示す
- 操作: `Copy-Item .env.example .env` 実行後の Explorer またはエディタを撮影
- 採用条件:
  - `.env` と `.env.example` の両方が識別できる
- 不採用条件:
  - ファイル名が見えない

### 02. `.env` 編集画面

- 目的: 起動に必要な設定値を確認したことを示す
- 操作: `.env` を開き、`HOST`、`PORT`、Provider API キー行、`ALLOW_EMAIL_LOGIN`、`ALLOW_REGISTRATION` が読める位置を撮影
- 採用条件:
  - 主要キー名が読める
  - 秘密値そのものは露出しない
- 不採用条件:
  - キー名が読めない
  - 秘密値をそのまま表示している

### 03. コンテナ起動確認

- 目的: LibreChat 関連サービスが起動したことを示す
- 操作: `docker compose ps` を撮影
- 採用条件:
  - `api` を含む主要サービスが `Up` と判断できる
  - 3080 番ポートが確認できる
- 不採用条件:
  - pull 中や create 中の途中状態だけ

### 04. サインアップ / ログイン画面

- 目的: UI に到達でき、利用開始可能であることを示す
- 操作: `http://localhost:3080` へアクセスし、サインアップまたはログイン画面を撮影
- 採用条件:
  - 入力欄と見出しが見える
  - ローディング中でない
- 不採用条件:
  - 502 / 503 / blank page

### 05. チャット入力画面

- 目的: 送信前にモデルと入力内容が確認できることを示す
- 操作:
  1. ログイン後、利用モデルを選択
  2. `こんにちは。3行で自己紹介して。` を入力
  3. 送信前に撮影する
- 採用条件:
  - モデル選択または Provider 文脈が見える
  - 入力文の主文が見える
  - 送信前である
- 不採用条件:
  - 入力欄が空
  - すでに応答が出ている

### 06. チャット出力画面

- 目的: 送信後に応答生成、または Tool Call / MCP 実行が成功したことを示す
- 操作:
  1. 05 の状態から送信する
  2. Tool Call / MCP を有効にした構成では、ツール名、実行ログ、結果のいずれかが見えるまで待つ
  3. Tool Call / MCP をまだ有効にしていない場合は、通常応答が見えるまで待つ
  3. 結果を撮影する
- 採用条件:
  - Tool Call / MCP を有効にした構成では、追跡可能なツール実行結果または失敗理由が見える
  - Tool Call / MCP 未実施の場合は、通常応答が見え、その理由を run-log に残している
  - 05 と異なる送信後状態である
- 不採用条件:
  - ローディング中
  - 05 と同一画面
  - 設定画面の代替

## Tool Call 分岐

- 本教材では、Agents / Tools / MCP を有効にした構成がある場合、その結果を 06 の第一候補とする
- 通常の OpenAI チャットで応答が返るだけの画面は、最小疎通確認としては有効だが、価値証跡としては優先度を下げる
- Tool Call / MCP を採用した場合は `run-log.txt` の備考欄に、使用した機能名、成功/失敗、結果の要点を記録する
- Tool Call / MCP を未実施で通常応答を採用した場合は、その理由と未実施範囲を `run-log.txt` に明記する

## 品質ゲート

1. 各画像の意味を 1 文で説明できること
2. 05 は送信前、06 は送信後であること
3. 主対象が画像中央付近にあり、余白が主役でないこと
4. Tool Call を使う場合は、結果追跡性があること
5. Tool Call / MCP を有効にした構成なら、06 は通常応答ではなく外部連携結果を優先していること
